## 1. 專案概述

### 1.1 背景與動機

在日常工作與學習中，人們經常面臨多項任務同時進行的情況，容易遺漏重要待辦事項或無法有效追蹤進度。現有的記事工具（如紙本筆記、簡單的備忘錄）缺乏結構化管理能力，無法提供任務狀態追蹤、優先順序排序等功能。

本專案旨在開發一個 **任務管理系統**，讓使用者能夠輕鬆地新增、編輯、刪除任務，並標記任務的完成狀態，從而提升個人或小型團隊的工作效率與任務可見度。

### 1.2 目標用戶

| 用戶類型 | 描述 |
|---------|------|
| **個人使用者** | 需要管理日常待辦事項的學生、上班族 |
| **小型團隊成員** | 需要追蹤團隊任務進度的專案成員 |
| **專案管理者** | 需要總覽所有任務狀態、分配工作的管理角色 |

### 1.3 核心價值主張

- **簡單直覺**：零學習成本，打開即可使用
- **即時掌握進度**：一目了然所有任務的完成狀態
- **彈性分類**：透過分類與優先順序，讓重要任務不被遺漏
- **隨時可用**：基於 Web 的應用，任何裝置皆可存取

## 2. 功能需求

### 2.1 主要功能列表

| 編號 | 功能名稱 | 說明 |
|------|---------|------|
| F1 | 任務 CRUD | 新增、檢視、編輯、刪除任務 |
| F2 | 完成狀態標記 | 將任務標記為「完成」或「未完成」 |
| F3 | 任務分類管理 | 建立分類（如：工作、學習、生活），並將任務歸類 |
| F4 | 優先順序設定 | 為任務設定高 / 中 / 低優先等級 |
| F5 | 到期日設定與提醒 | 為任務設定截止日期，並以視覺方式提示即將到期或逾期任務 |
| F6 | 任務搜尋與篩選 | 依關鍵字、分類、狀態、優先順序篩選任務 |
| F7 | 使用者帳號系統 | 註冊、登入、登出功能，每位使用者擁有獨立的任務清單 |

### 2.2 使用者故事（User Stories）

#### F1 — 任務 CRUD
- 作為 **使用者**，我希望 **能夠新增一筆任務並填寫標題與描述**，以便 **記錄需要完成的工作**。
- 作為 **使用者**，我希望 **能夠檢視我的所有任務列表**，以便 **掌握目前有哪些事情要做**。
- 作為 **使用者**，我希望 **能夠編輯已建立的任務內容**，以便 **修正錯誤或更新任務資訊**。
- 作為 **使用者**，我希望 **能夠刪除不再需要的任務**，以便 **保持任務清單的整潔**。

#### F2 — 完成狀態標記
- 作為 **使用者**，我希望 **能夠一鍵將任務標記為完成**，以便 **快速記錄進度**。
- 作為 **使用者**，我希望 **能夠將已完成的任務改回未完成**，以便 **處理誤按或需要重新執行的情況**。

#### F3 — 任務分類管理
- 作為 **使用者**，我希望 **能夠建立自訂分類**，以便 **依照不同情境整理任務**。
- 作為 **使用者**，我希望 **能夠為每筆任務指定所屬分類**，以便 **快速找到特定類型的任務**。

#### F4 — 優先順序設定
- 作為 **使用者**，我希望 **能夠為任務設定優先等級（高／中／低）**，以便 **優先處理最重要的任務**。
- 作為 **使用者**，我希望 **任務列表能依照優先順序排列**，以便 **一眼看出最緊急的事項**。

#### F5 — 到期日設定與提醒
- 作為 **使用者**，我希望 **能夠為任務設定截止日期**，以便 **在期限前完成任務**。
- 作為 **使用者**，我希望 **系統能以顏色或圖示標示即將到期或已逾期的任務**，以便 **不會錯過期限**。

#### F6 — 任務搜尋與篩選
- 作為 **使用者**，我希望 **能夠用關鍵字搜尋任務**，以便 **快速找到特定任務**。
- 作為 **使用者**，我希望 **能夠依分類、狀態、優先順序篩選任務**，以便 **專注於特定類型的工作**。

#### F7 — 使用者帳號系統
- 作為 **使用者**，我希望 **能夠註冊帳號並登入系統**，以便 **擁有專屬的任務空間**。
- 作為 **使用者**，我希望 **我的任務資料受到保護，只有我自己能存取**，以便 **確保隱私安全**。

## 3. 非功能需求

### 3.1 技術限制

| 項目 | 規格 |
|------|------|
| 後端框架 | Flask (Python) |
| 模板引擎 | Jinja2 |
| 資料庫 | SQLite |
| 前端 | HTML5 + CSS3 + JavaScript |
| 版本控制 | Git |

### 3.2 效能考量

- 頁面載入時間應在 **2 秒以內**
- 單一使用者的任務數量上限建議 **1,000 筆**，以確保查詢效能
- SQLite 適用於小型至中型規模，若未來需擴展可考慮遷移至 PostgreSQL

### 3.3 安全考量

- 使用者密碼需經過 **雜湊加密**（如 bcrypt）後儲存，禁止明文存放
- 所有表單需加入 **CSRF 防護**
- 使用者只能存取 **自己的任務資料**（權限隔離）
- 輸入欄位需進行 **基本驗證與過濾**，防止 XSS 與 SQL Injection

## 4. MVP 範圍

### Must Have（核心必備）

| 功能 | 說明 |
|------|------|
| 任務新增 | 使用者可以填寫標題、描述來建立任務 |
| 任務列表 | 顯示所有任務，含標題、狀態、建立時間 |
| 任務編輯 | 修改任務的標題與描述 |
| 任務刪除 | 刪除不需要的任務 |
| 完成狀態切換 | 一鍵切換任務的完成 / 未完成狀態 |
| 使用者註冊與登入 | 基本帳號系統，保護個人資料 |

### Should Have（重要但非首要）

| 功能 | 說明 |
|------|------|
| 任務分類 | 建立分類並將任務歸類 |
| 優先順序設定 | 為任務標記高 / 中 / 低優先等級 |
| 到期日設定 | 設定任務截止日期 |
| 逾期提示 | 以視覺方式標示逾期或即將到期的任務 |

### Nice to Have（加值功能）

| 功能 | 說明 |
|------|------|
| 搜尋功能 | 依關鍵字搜尋任務 |
| 進階篩選 | 依多條件組合篩選任務列表 |
| 任務排序 | 依優先順序、到期日、建立時間等排序 |
| 深色模式 | 提供暗色主題切換 |
| 批次操作 | 一次選取多筆任務進行刪除或狀態切換 |
